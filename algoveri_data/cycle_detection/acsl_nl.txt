Your task is to implement an algorithm that detects whether a directed unweighted graph contains a cycle and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: The graph is given as a CSR-encoded `Graph` struct with `n` vertices, a `row_ptr` array of length `n + 1`, and a `col_idx` array of length `row_ptr[n]`; the predicate `well_formed(g)` requires `row_ptr[0] == 0`, `row_ptr` monotone non-decreasing, every neighbour index in `[0, n)`, and the appropriate `\valid_read` ranges. Requirements: Implement `int has_cycle(const Graph *g)` returning `0` or `1`, with `(\result != 0) <==> graph_has_cycle(g)`. The natural algorithm is iterative DFS with three states per vertex (unvisited, in-stack, done): a back-edge to an in-stack vertex witnesses a cycle; the search terminating with no back-edge witnesses acyclicity. Verification Challenges: (1) Soundness: if you return `1` you must exhibit a path `p[0..plen)` with `is_path(g, p, plen)`, `plen > 1`, and `p[0] == p[plen - 1]`. The simplest witness is the current DFS stack closed back to the back-edge target. (2) Completeness: if you return `0` no such cycle exists in the entire graph; the strong invariant is that fully processed (black) vertices appear in a topological order, so any cycle would force a back-edge that the search would have detected. (3) Termination: even on cyclic graphs the visited-state trinary marking ensures each vertex is visited at most twice; supply a loop variant such as `n - (visited_count)` or sum over remaining work. ACSL-specific notes: (a) `graph_has_cycle` is defined as `\exists int *p, integer plen; ... is_cycle(g, p, plen)`; under the default `Typed` model WP treats this as quantification over phantom buffers, so the witness path must be a ghost array allocated inline (or simulated with an integer-indexed logic function). (b) The function is pure: `assigns \nothing`. The DFS internal state lives in local stack/visited arrays — model these with ghost arrays so the assigns clause does not have to mention them. (c) Mind the cast: `g->n` is `size_t`; when you write `0 <= v < g->n` ACSL widens through `integer`, but the inner `\exists integer e; g->row_ptr[u] <= e < g->row_ptr[u + 1]` references `g->row_ptr[u + 1]` which is also `size_t` — keep arithmetic in `integer` to avoid sign confusion. (d) Frama-C/WP discharges existential reachability queries best when you provide explicit ghost `instantiate` axioms or constructively provide the witness path as a concrete array; avoid relying on the SMT to invent paths. (e) Do not introduce top-level lemmas; the `well_formed`, `has_edge`, `is_path`, `is_cycle`, and `graph_has_cycle` predicates in the preamble are sufficient.
