Your task is to implement the `pop` operation for a Last-In-First-Out stack and verify its correctness in ACSL using Frama-C/WP. The stack is represented in C as three arguments: `int32_t *data` for the underlying buffer of capacity `cap`, `size_t *len` for the current length pointer, and `cap` for the maximum capacity; the live view is `data[0 .. *len - 1]` with the top at index `*len - 1`. Signature: `int32_t stack_pop(int32_t *data, size_t *len, size_t cap)`. Preconditions: `\valid(len)`, `0 < *len <= cap` (non-empty stack), and `\valid(data + (0 .. cap - 1))`. The contract specifies `assigns *len;` (the buffer is logically shrunk by one without overwriting the popped slot — the standard convention for an array-backed stack pop), and three postconditions: (1) the length decreases by one (`*len == \old(*len) - 1`), (2) the result equals the pre-state top (`\result == \at(data[\old(*len) - 1], Pre)`), and (3) the surviving prefix is unchanged (`\forall i; 0 <= i < *len ==> data[i] == \at(data[i], Pre)`). The implementation is constant-time: read `data[*len - 1]` into a temporary, decrement `*len`, return the temporary. Verification challenges: (1) Frame on `data` — although `data` is in the read frame and `assigns *len` excludes it, the third postcondition explicitly relates post-state cells to pre-state cells; WP discharges this from the `assigns` clause once aliasing is ruled out. (2) Aliasing between `len` and `data` — without separation, WP cannot conclude that decrementing `*len` does not change any `data[k]`. (3) Bounds — `*len > 0` is given, so `*len - 1` is a valid `size_t` index. ACSL-specific notes: (a) The contract requires `\separated(len, data + (0 .. cap - 1))` to rule out aliasing between the length pointer and the data buffer; without it the postcondition `data[i] == \at(data[i], Pre)` cannot be proved under `-wp-model Typed`. Verus's `&mut` borrow checker provides this guarantee implicitly; in ACSL it must be stated. (b) Use default `-wp-model Typed`; no axiomatic preamble is required. (c) `size_t` arithmetic: `*len - 1` is safe because `*len > 0`. (d) No top-level lemmas are required; the function has no loop, so no loop variant. (e) The result type `int32_t` and buffer element type `int32_t` agree; no implicit conversion concerns. (f) The contract intentionally does not specify the value of the popped slot post-pop; the standard convention is that the slot becomes "logically free" and may be written by future pushes — the third postcondition only constrains the live prefix `[0, *len)`.
