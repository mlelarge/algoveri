Your task is to implement the naive (linear) integer exponentiation algorithm and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: `b`, `e` are `uint64_t` and the mathematical result `spec_pow(b, e)` fits in `uint64_t` (i.e. `<= ULLONG_MAX`). Requirements: Implement `uint64_t exponentiation(uint64_t b, uint64_t e)` returning `b^e` by computing `result = 1; for i in 0..e: result *= b;` and prove `\result == spec_pow(b, e)`. Verification Challenges: (1) Accumulation Invariant: the loop invariant is `result == spec_pow(b, i)` where `i` is the loop counter; this falls out of `spec_pow_step` via `result_after = b * result_before == b * spec_pow(b, i) == spec_pow(b, i + 1)`. (2) Overflow Safety (Monotonicity): each multiplication `result *= b` must not overflow `uint64_t`. Since the precondition only bounds the *final* value `spec_pow(b, e)`, you must establish the lemma "if `0 <= i <= e` and `b >= 1` then `spec_pow(b, i) <= spec_pow(b, e)`" inline (case-split on `b == 0` and `b >= 1`). The case `b == 0` requires special handling because `0^0 = 1` then `0^k = 0` for `k >= 1`. (3) Boundary cases: `e == 0` returns `1` directly. ACSL-specific notes: (a) The axiomatic `IntegerExp` defines `spec_pow(b, 0) == 1` and `spec_pow_step` for `e > 0`; lift these as ghost asserts inside the loop rather than via top-level lemmas. (b) The contract uses `assigns \nothing` (pure function). (c) `b >= 0` and `e >= 0` are vacuous on `uint64_t` and serve only as documentation. (d) Use `loop variant e - i;` for termination. (e) The monotonicity argument needs the inline lemma `\forall integer b, i, e; 0 <= i <= e && b >= 1 ==> spec_pow(b, i) <= spec_pow(b, e)`; encode this as a recursive ghost function with `decreases e - i` rather than a top-level lemma. For `b == 0` either special-case at the top of the function or prove the analogous monotonicity via a small case-split. (f) Default `-wp-model Typed` is sufficient; with Alt-Ergo/Z3 the only painful obligations are the non-linear monotonicity lemma applications, which the inline ghost lemma resolves. (g) Do not add top-level lemmas — keep all assistance in inline ghost statements/functions.
