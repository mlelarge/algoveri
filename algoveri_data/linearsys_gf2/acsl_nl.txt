Your task is to implement a linear system solver over the field GF(2) (integers mod 2) and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: You are given a row-major flattened binary matrix `a` of dimensions `rows x cols` (with `0 < rows, cols <= 100`) and a binary right-hand-side vector `b` of length `rows`; both `a` and `b` contain only 0/1 entries (enforced by the predicates `is_binary_matrix` / `is_binary_vector`). The output buffer `x_out` is a pre-allocated array of `cols` `uint64_t`. Requirements: Implement `solve_linear_system_gf2` returning `int` (`1` on Some, `0` on None). (1) Algorithm: Standard Gaussian elimination over GF(2): row operations are XOR (addition in GF(2)) and the pivot-selection step swaps rows when the diagonal is zero. After forward elimination, perform back-substitution to recover `x_out`. Free variables (columns without a pivot) can be set to 0; if any all-zero pivot row has a non-zero RHS, return None (`0`). (2) Functional Correctness: `is_solution(a, b, rows, cols, x)` requires `x` is binary and that for every row `i`, the GF(2) dot product `dot_product_gf2(a, x, cols, i, cols-1) == b[i]`. The `dot_product_gf2` axiom block defines the dot product recursively from the right, with base, step, and skip axioms. The contract requires: when `\result == 1`, `is_solution(a, b, rows, cols, x_out)` holds; when `\result == 0`, no binary `x` satisfies `is_solution(a, b, rows, cols, x)`. Verification Challenges: (1) Solution-Set Preservation Under Row Operations: You must prove that swapping two rows and XOR-ing one row into another do not change the solution set. This requires a key lemma: if `dot_product_gf2(a, x, cols, i, cols-1) == b[i]` for the original pair `(a_i, b_i)`, the same holds for the transformed pair after a row operation. The proof needs distributivity of the dot product over GF(2) addition - usually best phrased as an inductive lemma on `k`. (2) Completeness of the None Case (Research-Grade): Proving `\result == 0 ==> \forall x. is_binary_vector(x, cols) ==> !is_solution(...)` is genuinely hard in any tool - it requires a formal argument that an inconsistent reduced system has no solution, which entails representing rank, span, or echelon form in ACSL logic. This is at the frontier of what is currently feasible with WP; an explicit ghost variable tracking the row-equivalence class plus auxiliary lemmas about the kernel of the reduced matrix is one viable but heavyweight strategy. Plan to scaffold this with several intermediate `assert` statements and possibly admit certain sub-lemmas. ACSL-specific notes: (a) The contract uses `assigns x_out[0 .. cols - 1];` so `\separated(x_out + (0 .. cols - 1), a + (0 .. rows*cols - 1))` is implicit but must be ensured by the caller; consider adding `requires \separated(x_out + (0..cols-1), a + (0..rows*cols-1));` and `requires \separated(x_out + (0..cols-1), b + (0..rows-1));` for clean verification. (b) Working over `uint64_t` rather than `bool` is intentional: it lets the C code use bitwise XOR (`^`) directly on the 0/1 values without bit-packing. (c) `-wp-model Typed` is sufficient; the `% 2` in `add_gf2` and `mul_gf2` is on mathematical integers (not modular C arithmetic), which Alt-Ergo handles natively. (d) No top-level lemmas are required by the contract, but the row-operation-preserves-solution lemma is essentially unavoidable as ghost-only auxiliary. (e) Modular arithmetic: since all values are in `{0,1}`, `(a+b) % 2` matches the XOR of `(uint64_t)(a^b)` for 1-bit values; this equivalence is a small lemma. (f) Modify scratch matrix copies on the stack; if you allocate them on the heap you must add appropriate `\separated` clauses.
