Your task is to implement an Optimized Trial-Division Primality Test (stop at sqrt(n)) and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: The input n is a uint64_t. Requirements: Implement the function `check_prime` which returns 1 if n is prime and 0 otherwise. (1) Algorithm: Iterate i from 2 while i*i <= n; as soon as i divides n return 0. After the loop, return 1 when n > 1 and 0 when n <= 1. The early termination at sqrt(n) is the key optimization. (2) Functional Correctness: You must verify that the result matches the mathematical predicate `is_prime(n) == (n > 1 && \forall integer d; 1 < d < n ==> !divides(d, n))` from the preamble, where `divides(d, n) == (d != 0 && n % d == 0)`. The hard part is bridging "no divisor up to sqrt(n)" with "no divisor up to n - 1". Verification Challenges: (1) Non-linear loop bound: the test `i * i <= n` is non-linear; SMT solvers (Alt-Ergo, Z3, CVC4) frequently time out on such guards. The loop invariant should be stated using a linear bound (e.g., `i <= n` or `i - 1 <= isqrt(n)`) and the multiplication only appears in the loop condition. (2) Cofactor lemma: To discharge the post-condition you must invoke a number-theoretic lemma — if n is composite then it has a divisor d with 2 <= d <= sqrt(n) (equivalently, d*d <= n). Concretely: if d divides n with 1 < d < n then either d <= n/d, in which case d*d <= n, or n/d <= d, in which case the cofactor d' = n/d satisfies 1 < d' <= sqrt(n) and divides n. Stating this as a lemma at the WP level is SMT-hard because of the nonlinear `cof_factor_lt2`-style argument; you may need a Coq-discharged lemma or a ghost-step that exhibits the cofactor explicitly. (3) Loop invariant: maintain `\forall integer d; 1 < d < i ==> !divides(d, n)` plus `i * i <= n + (i + 1)` or similar to keep arithmetic linear. ACSL-specific notes: (a) The function is pure: `assigns \nothing;`. (b) Provide `loop variant n - i;` (or a square-bracketed variant) for termination since the loop bound depends on the squared comparison. (c) Use `-wp-model Typed`; uint64_t multiplication can overflow when i*i > 2^64, so guard with `i <= UINT32_MAX` or test `i <= n / i` instead — the spec is correct but the implementation must dodge overflow. (d) The cofactor lemma is the SMT-hardest fact in the whole benchmark; expect to need an explicit `lemma` block or a ghost witness. The base predicates `divides` and `is_prime` in the preamble are sufficient — no top-level axioms beyond them.
