Your task is to implement and verify the deletion algorithm for a Left-Leaning Red-Black Tree (LLRBT) in ACSL using Frama-C/WP. Preconditions: `t` is either `\null` or a valid `LlrbtNode *` satisfying `is_llrbt(t)`, captured by the helper predicate `opt_is_llrbt`. Requirements: Implement `llrbt_delete` returning an `LlrbtNode *` (possibly `\null` if the resulting set is empty). (1) Algorithm: The classical recursive deletion with red-link maintenance: descend to the target value while ensuring the current node is part of a "3-node" cluster (Red node or has a Red left child), pushing reds down the search path via `move_red_left` and `move_red_right` helpers; when the target is found, replace it with its in-order successor (or simply detach if it is a leaf in the rightmost position). After unwinding, re-balance with the standard LLRBT fixup sequence (rotate left if right child is Red, rotate right if left and left.left are Red, flip colours if both children are Red). (2) Functional Correctness: The contract requires that when `\result != \null`, `is_llrbt(\result)` holds and `in_view(\result, x) <==> (t != \null && in_view{Pre}(t, x) && x != v)` (the new view is the old view minus `v`); when `\result == \null`, either `t` was originally `\null` or the original view contained at most the value `v`. STRUCTURAL DEFECT (flagged challenge): The contract states `assigns \nothing;` while requiring substantial structural change to the tree (red-link rotations, recolouring, node detachment). These are jointly unsatisfiable: any non-trivial deletion either mutates pointer fields or allocates fresh nodes, both of which violate `assigns \nothing;`. The recommended fix is identical in spirit to `llrbt_insert`: weaken `assigns` to cover the entire reachable footprint of `t`, or treat the algorithm as functionally allocating with `allocates \fresh; frees \old_tree;`. The latter is more honest to the Verus original (`fn delete(tree: Option<Box<Node>>) -> Option<Box<Node>>` operates on owned values) but is awkward to write in pure C+ACSL. As written, the spec is unsatisfiable. Verification Challenges: (1) Structural Defect (above) - blocks any sound implementation. (2) `move_red_left` / `move_red_right` Helpers: These are the technical heart of LLRBT deletion. Each requires its own contract proving "if the current path needs a Red, this primitive supplies one without violating black-height"; both must be verified before delete itself can be discharged. (3) Successor-Replacement Subroutine: The case where the target has two non-null children involves splicing the in-order successor's value into the target node and recursively deleting the successor; the recursion uses the same delete contract on a strictly smaller tree (the successor is in `t->right`'s subtree). (4) Termination: The recursive structure has multiple recursive calls (one per child plus the successor-deletion call); a structural variant on tree size (number of nodes, available as a ghost logic function) is required to prove termination. (5) Black-Height Preservation: Like insert, delete must preserve `black_height` exactly. The proof composes the contracts of `move_red_left`/`right` (each preserves the height) with the rotation contracts. ACSL-specific notes: (a) The `assigns \nothing;` clause is the load-bearing defect; weakening it (or admitting a fresh-allocation interpretation) is a prerequisite for any sound implementation. (b) The recursion depth is `O(log n)`, but ACSL imposes no bound; a ghost variant is required for termination. (c) `-wp-model Typed` is sufficient. (d) No top-level lemmas required by contract, but at minimum: `move_red_left` and `move_red_right` correctness lemmas, plus the three rotation/flip primitives, plus a height-preservation lemma per primitive - this is a substantial proof investment. (e) Aliasing: ACSL has no built-in acyclicity for tree structures; ghost separation across all subtrees of the reachable footprint is needed for any in-place implementation.
