Your task is to implement Binary Search Tree deletion and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: The input is a (possibly null) pointer `t` to a `BstNode` satisfying `opt_is_bst(t)`. The value to remove is a `uint64_t v`. Requirements: Implement `bst_delete(BstNode *t, uint64_t v)` returning `BstNode *` — possibly `NULL` (if the resulting set is empty), otherwise the root of the new BST. (1) Algorithm: Use the standard recursive scheme. If `t == NULL` return `NULL`. Otherwise, if `v < t->val`, recurse into the left subtree and re-attach. If `v > t->val`, recurse into the right subtree and re-attach. If `v == t->val`: (a) if both children are NULL, return NULL; (b) if only one is NULL, return the other; (c) otherwise, find the in-order successor (leftmost descendant of `t->right`), copy its value into `t`, and delete it from `t->right`. Return the resulting root. (2) Functional Correctness: Two postconditions, indexed by whether `\result` is null. If `\result != \null`, then `is_bst(\result)` and `\forall integer x; in_view(\result, x) <==> (t != \null && in_view{Pre}(t, x) && x != v)`. If `\result == \null`, then `t == \null || \forall integer x; in_view{Pre}(t, x) ==> x == v` (the original tree was empty or contained only `v`'s — single occurrence in a BST). The preamble provides `in_view`, `is_bst`, and `opt_is_bst` (recursive predicates). Verification Challenges: (1) BST invariant preservation: re-attachment after a recursive delete must respect the parent's ordering. The successor-replacement case is the trickiest: after copying the successor value into `t` and removing it from the right subtree, you must show that the new `t->val` is still `> y` for every `y` in `t->left` (this follows because the old `t->val` was `> y` and the successor is the *minimum* of the right subtree, hence `> t->val_old > y`). (2) View update: the resulting view equals the old view minus `v`. The successor case requires showing that the successor's old position no longer contributes to the view, but its value now appears at the root, so net change is "remove `v`, since the BST invariant guarantees `v` had at most one occurrence". (3) Termination: each recursive call decreases the structural size of the tree. ACSL-specific notes: (a) The contract is `assigns \nothing;` — as in `bst_insert`, the implementation allocates fresh nodes (or carefully reuses existing ones via the recursive functional pattern); writes to *fresh* memory are not constrained by `assigns`. If your implementation mutates fields of the input nodes in place (e.g., overwriting `t->val` during the successor-replacement step), you must instead change `assigns \nothing` to an explicit footprint such as `assigns t->val, t->left, t->right, ...` — but the provided spec's `assigns \nothing` corresponds to the purely-functional rebuild style, which is what we recommend. (b) `in_view{Pre}` evaluates at the `Pre` label so the postcondition refers to the original tree. (c) `in_view` and `is_bst` are recursive predicates; provide a `decreases` measure (e.g., a ghost-tracked tree height) for recursive calls. (d) The default `-wp-model Typed` memory model is sufficient. (e) The successor-replacement step is best factored into a helper function `bst_remove_min(BstNode *t, uint64_t *out_min)` with its own contract — keep that helper *inside* the same translation unit and document it via `requires`/`ensures` clauses, not via top-level lemmas. (f) Per project rules, no top-level lemmas or axioms in the user solution; encode the BST-minimum and view-update arguments as `assert` chains within the implementation.
