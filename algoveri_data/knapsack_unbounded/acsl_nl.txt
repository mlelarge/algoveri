Your task is to implement the Unbounded (Complete) Knapsack algorithm (each item may be chosen any non-negative number of times) and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: You are given `const uint64_t *` arrays `weights` and `values` of length `n` with `0 < n <= 100`; per-item weights are strictly positive (`0 < weights[i] <= 1000`) which prevents zero-weight items inflating value indefinitely; per-item values are bounded by 1000; capacity `capacity <= 1000`. Both arrays must be read-valid over `0 .. n - 1`. Requirements: Implement `solve_knapsack_unbounded` returning a `uint64_t` equal to the optimal achievable value. (1) Algorithm: Use the standard 1-D DP `dp[w]` of length `capacity + 1` iterating capacity in ascending order so each item can contribute multiple times. (2) Functional Correctness: Strategies are represented as `int *counts` arrays whose entries must be non-negative (`counts[i] >= 0`); the axiomatic block defines `total_weight(counts, weights, k, n) = counts[k] * weights[k] + total_weight(.., k+1, ..)` and similarly for `total_value`. You must prove Upper Bound (every valid strategy yields value `<= \result`) and Existence (some valid strategy achieves `\result`). Verification Challenges: (1) Existence Witness: As with 0/1, the existence postcondition requires producing a counts vector via DP traceback - typically a ghost array filled while computing `dp[]`. (2) Upper Bound: The proof inducts on capacity rather than items: the loop invariant "for all valid `counts` with `total_weight <= w`, `total_value <= dp[w]`" is preserved when extending an extra item by one unit. The axiomatic step axiom must be instantiated with the chosen item's index. (3) Multiplicative term `counts[k] * weights[k]`: SMT solvers struggle with non-linear arithmetic; constraining `counts[k]` and `weights[k]` to small ranges (the precondition gives `weights[k] <= 1000`) and inducting on `counts[k]` rather than reasoning about the product directly is the standard workaround. ACSL-specific notes: (a) The contract is `assigns \nothing;` - all DP scratch is local. (b) No `\separated` clause is needed for the read-only inputs, but a DP buffer must be `\separated` from `weights` and `values` if you place it outside the stack. (c) `counts` is typed `int *` rather than `uint64_t *` because Verus's `Seq<int>` allows arbitrary integers including, in principle, negative ones; the `>= 0` constraint in `is_valid_strategy` is what makes the spec sound. (d) `-wp-model Typed` is sufficient; the multiplicative `counts[k] * weights[k]` term may require enabling Z3 or CVC4 for non-linear arithmetic, or auxiliary lemmas on `total_weight` monotonicity in `counts[k]`. (e) No top-level lemmas required, but a helper lemma `total_weight(c, w, k, n) >= 0 when all c[i] >= 0` is often useful for the upper-bound induction.
