Your task is to implement the "Exponentiation by Squaring" algorithm and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: `b`, `e` are `uint64_t` and the mathematical result `spec_pow(b, e)` fits in `uint64_t` (i.e. `<= ULLONG_MAX`). Requirements: Implement `uint64_t exponentiation(uint64_t b, uint64_t e)` returning `b^e` in `O(log e)` time using binary decomposition: maintain `result = 1`, `base = b`, `exp = e`, and on each iteration if `exp & 1` multiply `result *= base`, then `base *= base` and `exp >>= 1`. Verification Challenges: (1) Conservation Invariant: rather than the naive `result == b^i`, the loop invariant is the sophisticated `result * base^exp == b_orig^e_orig`. Establishing it at the bottom of each iteration uses the algebraic identities `(base^2)^(exp/2) == base^exp` (when `exp` is even) and `base * base^(exp - 1) == base^exp` (when `exp` is odd). (2) Non-Linear Arithmetic: SMT solvers struggle with `(b * b)^k == b^(2k)`; you will need explicit ghost lemmas (applied inline as `assert` statements that unfold `spec_pow_step` twice, or a recursive ghost function call) to convince WP. (3) Overflow Safety: each multiplication `result *= base` and `base *= base` must stay in `[0, ULLONG_MAX]`. The precondition `spec_pow(b, e) <= ULLONG_MAX` only bounds the final answer; for intermediate values you must additionally argue that whenever `base` is squared its squared value is itself bounded by some quantity reachable from `spec_pow(b_orig, e_orig)`, or guard the squaring with the standard "skip the final useless square" optimisation. ACSL-specific notes: (a) The axiomatic `FastExp` defines `spec_pow` recursively with `base` (`spec_pow(b, 0) == 1`) and `step` (`e > 0 ==> spec_pow(b, e) == b * spec_pow(b, e - 1)`); proofs need induction unrolled into loop invariants, not top-level lemmas. (b) Encode the conservation invariant as `loop invariant result * spec_pow(base, exp) == spec_pow(\at(b, Pre), \at(e, Pre));`. The `\at(., Pre)` references freeze the original arguments. (c) Use `loop variant exp;` for termination since `exp` strictly halves. (d) `assigns \nothing` because the function is pure. (e) Both `b >= 0` and `e >= 0` in the contract are vacuous for `uint64_t` and serve only as documentation. (f) Use the default `-wp-model Typed`; Alt-Ergo will need help on the non-linear step — supply per-branch asserts such as `//@ assert exp % 2 == 1 ==> spec_pow(base, exp) == base * spec_pow(base, exp - 1);` and lemma-style ghost functions that unfold `spec_pow(base*base, exp/2) == spec_pow(base, exp)`. (g) Do not add top-level lemmas; keep all assistance inline as ghost asserts.
