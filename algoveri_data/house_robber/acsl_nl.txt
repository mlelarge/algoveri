Your task is to implement the House Robber dynamic-programming algorithm and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: `nums` is `const uint64_t *` of length `n` with `0 <= n <= 100` and every `nums[i] <= 10000`. The signature is `uint64_t rob(const uint64_t *nums, int n)`. Requirements: Return the maximum total loot achievable by selecting a subset of indices that contains no two adjacent indices, where the loot is `\sum_i nums[chosen_i]`. The contract pins this down via two postconditions: an upper bound (`forall valid robbery, total_loot <= \result`) and existence (`exists valid robbery, total_loot == \result`), where `is_valid_robbery(houses, hlen, n)` requires every chosen index in `[0, n)` and consecutive choices to be at least two apart. Algorithm: classic DP with the recurrence `dp[i] = max(dp[i - 1], dp[i - 2] + nums[i])`, base cases `dp[-1] = 0`, `dp[0] = nums[0]`. Two scalar accumulators (`prev`, `prev2`) suffice. Verification Challenges: (1) Upper Bound: prove `forall houses, hlen; is_valid_robbery(houses, hlen, n) ==> total_loot(houses, 0, hlen, nums, n) <= dp[n - 1]`. The induction is on the largest chosen index: any valid robbery either skips index `n - 1` (bounded by `dp[n - 2]`) or uses it (bounded by `dp[n - 3] + nums[n - 1]`). (2) Existence: maintain a ghost trace of indices yielding `dp[i]` so you can return a concrete `int *houses` witness. (3) Loop Invariant: `prev == dp[i - 1]` and `prev2 == dp[i - 2]` at iteration `i`, plus the ghost-witness clauses for both bounds. ACSL-specific notes: (a) The axiomatic `HouseRobberSum` defines `total_loot(houses, k, hlen, nums, nums_len)` as `\sum_{j>=k} nums[houses[j]]` (with the in-range guard) using `total_loot_base` at `k == hlen` and `total_loot_step` for `0 <= k < hlen`. Drive proofs by inline ghost induction on `k`, not top-level lemmas. (b) The `is_valid_robbery` predicate guards `(hlen == 0 || \valid_read(houses + (0 .. hlen - 1)))`, so the empty robbery (`hlen == 0`) is always valid; expect WP to ask you to handle `n == 0` and `n == 1` separately. (c) Both ensures are existential/universal over `int *houses` and `integer hlen`; WP under the default `Typed` model treats these as quantification over phantom buffers, so existence must be supplied with a concrete (ghost) array filled in by the implementation. (d) `assigns \nothing` because the function is pure; ghost arrays are not mentioned in `assigns`. (e) Loot fits comfortably in `uint64_t` (`100 * 10000 = 10^6`). (f) Use a loop variant `n - i` for termination. (g) Do not introduce top-level lemmas; the axiomatic plus inline ghost code is sufficient.
