Your task is to implement the `push` operation for a Last-In-First-Out stack and verify its correctness in ACSL using Frama-C/WP. The stack is represented in C as three arguments plus the value to push: `int32_t *data` for the underlying buffer, `size_t *len` for the current length pointer, `size_t cap` for the maximum capacity, and `int32_t value` for the new element; the live view is `data[0 .. *len - 1]` with the top at index `*len - 1`. Signature: `void stack_push(int32_t *data, size_t *len, size_t cap, int32_t value)`. Preconditions: `\valid(len)`, `0 <= *len < cap` (room available), and `\valid(data + (0 .. cap - 1))`. The contract specifies `assigns data[*len], *len;` (writes the new element into slot `data[*len]`, then increments the length) and three postconditions: (1) the length increases by one (`*len == \old(*len) + 1`), (2) the surviving prefix is unchanged (`\forall i; 0 <= i < \old(*len) ==> data[i] == \at(data[i], Pre)`), and (3) the new element is at the correct slot (`data[\old(*len)] == value`). The implementation is constant-time: write `data[*len] = value;` then `(*len)++;`. Verification challenges: (1) Aliasing between `len` and `data[*len]` — when the implementation writes `data[*len] = value` it indirectly reads `*len`, and when it later writes `(*len)++` WP must know this does not invalidate the `data[\old(*len)] == value` postcondition. (2) Frame on the prefix — although `assigns data[*len]` is a single-cell write, WP discharges the prefix-unchanged postcondition automatically once aliasing between `len` and the prefix is ruled out. (3) Bounds — `*len < cap` is given, so `*len` is a valid index into `data[0 .. cap - 1]`. ACSL-specific notes: (a) The contract requires `\separated(len, data + (0 .. cap - 1))` to rule out aliasing between the length pointer and the data buffer; without it WP cannot prove that writing `data[*len]` does not also modify `*len`, nor that incrementing `*len` does not perturb `data[\old(*len)]`. Verus's `&mut` borrow checker provides this implicitly; in ACSL it must be stated. (b) Use default `-wp-model Typed`; no axiomatic preamble is required. (c) `size_t` arithmetic: `*len + 1 <= cap` follows from `*len < cap`, so the increment is safe. (d) No top-level lemmas are required; the function has no loop, so no loop variant. (e) The element type `int32_t` matches the value parameter; no implicit conversion. (f) `assigns data[*len]` uses the pre-state value of `*len` (per ACSL semantics), which is exactly the slot to be written; this clause must be retained even if you implement the push differently — for instance, incrementing first and writing `data[*len - 1] = value` would change the assigns clause to `data[*len - 1]` after increment, which is equivalent.
