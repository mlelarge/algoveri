Your task is to implement Kadane's algorithm for the Maximum Subarray Sum problem and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: the input is `int32_t *a` of length n with 0 < n <= 100,000, and `\valid_read(a + (0 .. n - 1))`. Requirements: implement `max_subarray_sum(const int32_t *a, size_t n)` returning an int64_t equal to the maximum sum of any contiguous subarray (including the empty subarray, which has sum 0). (1) Algorithm: Kadane's classic single-pass — maintain a running variable `cur` (max sum of a subarray ending at the current index) and a running maximum `best`; at each step set `cur = max(0, cur + a[i])` (or `cur = max(a[i], cur + a[i])` for non-empty version) and `best = max(best, cur)`. Return `best`. (2) Functional Correctness: prove that for every valid range `0 <= i <= j <= n`, `spec_sum(a, i, j) <= \result` (universal upper bound), and that there exists at least one such (i, j) with `spec_sum(a, i, j) == \result` (witness existence). The `spec_sum` logic function is defined recursively in the preamble: `spec_sum(a, start, end) = (start >= end) ? 0 : a[end - 1] + spec_sum(a, start, end - 1)`. Verification Challenges: (1) Loop invariant — at the end of iteration i, `best` must equal the max of `spec_sum(a, p, q)` over `0 <= p <= q <= i + 1`, and `cur` must equal the max of `spec_sum(a, p, i + 1)` over `0 <= p <= i + 1`. Stating both inductively is the heart of Kadane's verification. (2) Witness existence — to discharge the existential postcondition, you must record the (best_start, best_end) pair as ghost variables updated whenever `best` is updated. (3) Overflow safety — using int64_t for the accumulator is essential because n * INT32_MIN can underflow int32_t; the bound n <= 100_000 keeps cur and best inside [-100_000 * 2^31, 100_000 * 2^31] which fits in int64_t. ACSL-specific notes: (a) The contract uses `assigns \nothing;` since the function is pure on its input; no `\separated` clause is required because no pointer is written through. (b) Add a `loop variant n - i` for termination. (c) `spec_sum` is expressed as a recursive `logic integer` rather than an axiomatic block — Frama-C/WP will unfold it on demand, so no top-level lemmas are needed. (d) `-wp-model Typed` is the right model; the int64_t accumulator should be reasoned about as machine integers, and the bound proof must explicitly assert |cur| <= n * INT32_MAX inside the invariant.
