Your task is to implement a naive solver for the Discrete Logarithm Problem and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: `g`, `h`, `p` are `uint64_t` and `p > 1`; the caller provides `uint64_t *out_x` for the result discriminator. Requirements: Implement `int discrete_log_naive(uint64_t g, uint64_t h, uint64_t p, uint64_t *out_x)`. Return `1` (Some) and write `0 <= *out_x < p` with `is_discrete_log(g, h, p, *out_x)` and `*out_x` minimal, or return `0` (None) if no `x` in `[0, p)` satisfies `g^x \equiv h \pmod p`. Algorithm: Iterate `x` from `0` to `p - 1` while keeping `current_val = g^x mod p` updated incrementally as `current_val = (current_val * g) % p`, returning Some(x) on the first match against `h % p` and None on loop exhaustion. Verification Challenges: (1) Incremental Modular Exponentiation: Prove the loop invariant `current_val == spec_pow_mod(g, x, p)`. The base case at `x = 0` is `spec_pow_mod_base` (the value `1`); the step uses `spec_pow_mod_step` plus the modular-multiplication identity `(a * b) mod p == ((a mod p) * b) mod p`. (2) Negative Proof (None): if the loop exits, every `k` in `[0, p)` failed the test; this requires the invariant `\forall integer k; 0 <= k < x ==> !is_discrete_log(g, h, p, k)`. (3) Minimality (Some): when you return `Some(x)`, the same invariant gives the "no earlier solution" clause for free. (4) Overflow: `(current_val * g) % p` with `p` up to `2^64 - 1` can overflow `uint64_t`; rather than relying on `__int128` you can require a smaller `p` bound or implement Russian-peasant modular multiplication. The current contract does not bound `p` further, so the implementation should choose a representation that the WP can prove safe (commonly: `p <= 2^32` precondition added by you, or a verified mulmod helper). ACSL-specific notes: (a) The axiomatic `DiscreteLog` defines `spec_pow_mod` with two axioms (`base`, `step`); proofs proceed by induction unrolled into the loop invariant, not via top-level lemmas. (b) ACSL's `%` on `integer` agrees with the C operator on non-negative operands; here all of `g`, `h`, `p`, `current_val` are non-negative so spec-side and code-side `%` coincide. Beware: ACSL `%` is C-truncating, *not* Euclidean, so any extra lemma of the form `0 <= x < 2 * p ==> (x - p) % p == x % p` is *false* for the lower half (when `0 <= x < p`, `x - p` is negative and the truncating `%` returns a non-negative residue different from `x`). Do not introduce such a lemma. (c) `assigns *out_x` is the only write. (d) Use a loop variant `p - x` for termination. (e) Default `-wp-model Typed` plus Alt-Ergo/Z3 should suffice; supply `assert` statements inside the loop to instantiate the `spec_pow_mod_step` axiom at the current value of `x`. No top-level lemmas should be added.
