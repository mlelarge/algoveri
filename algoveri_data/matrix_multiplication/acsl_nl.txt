Your task is to implement Matrix Multiplication and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: the inputs a and b are flat row-major uint64_t arrays representing matrices of shape rows x mid and mid x cols respectively; the output c is a uint64_t array of shape rows x cols. The dimensions satisfy 0 < rows, mid, cols <= 10 (so rows*cols and rows*mid stay tiny), and every entry of a and b lies in [0, 100], which keeps each dot product strictly under 10 * 100 * 100 = 100_000 and well within uint64_t. Requirements: implement the function `matrix_multiply` so that c[r * cols + col] equals the dot product of row r of a with column col of b for all r in [0, rows) and col in [0, cols). (1) Algorithm: the canonical triple-nested loop — for each (r, col), accumulate sum += a[r * mid + k] * b[k * cols + col] for k in [0, mid). (2) Functional Correctness: the postcondition uses the axiomatic logic function `dot_product(a, b, mid, cols, r, c, k)` whose base case (k <= 0) is 0 and whose step case is a[r*mid + (k-1)] * b[(k-1)*cols + c] + dot_product(.., k-1). You must prove that, after termination, c[r*cols + col] == dot_product(a, b, mid, cols, r, col, mid) for every valid (r, col). Verification Challenges: (1) Inner-loop invariant — at iteration k of the innermost loop you must state that the running accumulator equals dot_product(a, b, mid, cols, r, col, k) and lift this to the postcondition when k == mid. (2) Frame conditions for partial writes — the outer two loops only update a contiguous slice of c, so the loop invariants must record which cells of c are already final and which still hold their initial value. (3) Overflow safety — although the problem constants make overflow impossible, the proof obligation must explicitly bound the partial sum (<= mid * 10000) to discharge the uint64_t multiplication and addition checks. ACSL-specific notes: (a) The contract requires `\separated(c + (0 .. rows*cols - 1), a + (0 .. rows*mid - 1), b + (0 .. mid*cols - 1))` so that writes to c never invalidate the predicate reads on a and b — without this the dot_product reads are not framed across the inner update. (b) The `reads` clause of dot_product currently lists `a[0 .. r*mid + mid - 1], b[0 .. (mid-1)*cols + c]`; you may need to widen these to the full input slabs so the `assigns c[..]` clause discharges automatically. (c) Use `loop variant` mid - k (resp. cols - col, rows - r) for termination of each loop. (d) `-wp-model Typed` plus the `\separated` clause is sufficient; no global lemmas are required.
