Your task is to implement Bubble Sort and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: The input array `a` of `int32_t` has length `n` (a `size_t`) with `0 <= n <= 1,000,000`; if `n > 0` the buffer `a + (0..n-1)` must be valid for both reading and writing. Requirements: Implement the function `bubble_sort(int32_t *a, size_t n)` which sorts the array in place. (1) Algorithm: Repeatedly pass through the array, swapping adjacent out-of-order pairs. After `k` outer iterations, the largest `k` elements occupy positions `n-k .. n-1` in sorted order. Stop after `n` outer passes (or earlier if a pass made no swaps). (2) Functional Correctness: The two postconditions are `is_sorted(a, n)` (every pair of indices is in order) and `same_multiset{Pre,Post}(a, n)` (the multiset of values is unchanged). The preamble defines: the predicate `is_sorted{L}(a, n) = \forall integer i, j; 0 <= i < j < n ==> a[i] <= a[j]`; the recursive logic function `count_occ{L}(a, n, x) = (n <= 0 ? 0 : count_occ(a, n - 1, x) + (a[n-1] == x ? 1 : 0))`; and the binary-label predicate `same_multiset{L1,L2}(a, n) = \forall integer x; count_occ{L1}(a, n, x) == count_occ{L2}(a, n, x)`. Verification Challenges: (1) Loop invariants for the two-level loop: after the outer loop runs `k` times, `a[n-k .. n-1]` is sorted and is greater than or equal to every element of `a[0 .. n-k-1]`; the inner loop maintains a similar invariant on the running maximum. (2) Permutation: prove that every swap is an exchange of two adjacent values, hence preserves the multiset. The multiset invariant `same_multiset{Pre, Here}(a, n)` should be maintained at the top of every iteration. (3) Termination: both loops are bounded by `n`. ACSL-specific notes: (a) `assigns a[0 .. n - 1];` is the only side effect, so no `\separated` clauses are required (only one buffer). (b) `count_occ` is recursive but is a logic function (not in an axiomatic block). Frama-C will accept it; for WP discharge you may need to introduce a small ghost lemma (proven by induction in a ghost function) connecting `count_occ` before and after a single adjacent swap — but per the project rule "no top-level lemmas or axioms in the user solution", you must encode this reasoning inline as `assert` chains and ghost-loop invariants instead. (c) Provide `loop variant n - i;` for the outer loop and `n - 1 - i - j;` for the inner loop. (d) The default `-wp-model Typed` memory model is sufficient. (e) The `int32_t` element type means signed comparison and copy; no overflow can occur since you only swap, not arithmetic. (f) The two-label predicate `same_multiset{Pre, Post}` evaluates `count_occ` at both labels — supplying `loop invariant same_multiset{Pre, Here}(a, n);` at every loop level is the cleanest discipline.
