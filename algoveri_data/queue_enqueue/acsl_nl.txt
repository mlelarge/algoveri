Your task is to implement the enqueue (push-back) operation of a verifiable Queue and verify its FIFO correctness in ACSL using Frama-C/WP. Preconditions: the queue is encoded by `int32_t *data`, `size_t *len`, and a fixed `cap`, with the live view being `data[0 .. *len - 1]`. The contract requires `\valid(len)`, `0 <= *len < cap`, and `\valid(data + (0 .. cap - 1))`. Requirements: implement `void queue_enqueue(int32_t *data, size_t *len, size_t cap, int32_t value)` so that the value is written at position *len and *len is incremented. (1) Algorithm: a single pair of writes — `data[*len] = value;` then `*len = *len + 1;`. (2) Functional Correctness: the contract states three obligations — `*len == \old(*len) + 1` (size increases by one), `\forall integer i; 0 <= i < \old(*len) ==> data[i] == \at(data[i], Pre)` (older elements preserved at their original indices), and `data[\old(*len)] == value` (the new element is appended at the back). Verification Challenges: (1) Frame condition — the `assigns data[*len], *len;` clause must be interpreted under the *post-state* value of *len; ACSL evaluates `data[*len]` here at the post-state, so the assigned cell is `data[\old(*len) + 1 - 1] == data[\old(*len)]`. Some users prefer to write `assigns data[\old(*len)], *len;` for clarity. (2) Aliasing — without `\separated`, the assignment to *len could in principle alias one of the `data[i]` cells, which would invalidate the "older elements preserved" postcondition. (3) Capacity bound — `*len < cap` ensures the write to `data[*len]` is within the validity range `\valid(data + (0 .. cap - 1))`. ACSL-specific notes: (a) The contract requires `\separated(data + (0 .. cap - 1), len)` so the increment of *len does not alias any data cell — Verus's `&mut data` and `&mut len` give this implicitly via the borrow-checker, but in ACSL it must be stated explicitly. (b) No loop is required; the function body is straight-line code, so no `loop variant` is needed. (c) The contract is otherwise self-contained; no axiomatic preamble is required because the queue's view is just `data[0 .. *len - 1]` and append-at-back is captured by the postcondition equations. (d) `-wp-model Typed` is sufficient; expect Alt-Ergo to discharge all three postconditions in one shot once `\separated` is in place.
