Your task is to implement and verify the dequeue (pop-front) operation for a fixed-capacity Ring Buffer in ACSL using Frama-C/WP. In C the buffer is represented as four arguments: `int32_t *data`, `size_t *head`, `size_t *len`, and `size_t cap`. The "live view" of the queue is `data[(*head + i) % cap]` for `0 <= i < *len`; the front is at index 0 of this view and is at memory cell `data[*head]`. Signature: `int32_t ringbuffer_dequeue(int32_t *data, size_t *head, size_t *len, size_t cap)`. Preconditions: `0 < cap <= 1024`, `data` is fully valid over `[0, cap-1]`, `head` and `len` are valid pointers, `0 <= *head < cap`, and `0 < *len <= cap` (non-empty). The contract specifies `assigns *head, *len;` (data is read-only on dequeue) and four postconditions: (1) the result equals the pre-state front `\at(data[*head], Pre)`, (2) the length decreases by one, (3) the head advances by one modulo cap, and (4) the remaining view shifts: for every `i` in `[0, *len)`, `data[(*head + i) % cap]` equals the pre-state cell `(\at(*head, Pre) + i + 1) % cap`. The implementation should advance the head and decrement the length in O(1), wrapping around at `cap`. Verification challenges: (1) Modular-arithmetic reasoning — Frama-C/WP must discharge `(h + 1) % cap` semantics; bounds `0 <= *head < cap` make this routine for Alt-Ergo. (2) Frame on the buffer cells — although `data` is not assigned, the postcondition relates post-state and pre-state cells through `\at`, and WP needs to know that `data[k]` is unchanged for every `k` (which follows from `assigns *head, *len`). (3) Aliasing — the head pointer, length pointer, and buffer cells must be disjoint or the post-state `data[(*head + i) % cap] == \at(data[(*head_pre + i + 1) % cap], Pre)` is unprovable. ACSL-specific notes: (a) The contract requires `\separated(head, len, data + (0 .. cap - 1))` to rule out aliasing between the metadata pointers and the data buffer — Verus's `&mut` borrow checker provides this implicitly, but in ACSL it must be stated. (b) Use the default `-wp-model Typed`; no axiomatic preamble is needed beyond standard pointer arithmetic. (c) `size_t` is treated as a non-negative machine integer, and `%` on non-negative size_t agrees with the mathematical modulo, so the modular arithmetic in the postconditions is sound. (d) No top-level lemmas are required. (e) The function has no loop, so no loop variant is needed; the body is a constant-time pointer update.
