Your task is to implement the standard O(N^2) algorithm for polynomial multiplication and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: inputs `a` and `b` are `int64_t` arrays of lengths na and nb respectively, both positive, with na + nb <= 1000, and every coefficient of each input bounded by `coeffs_bounded` to lie in [-1_000_000, 1_000_000]. The output buffer `out` has length na + nb - 1 with `\valid(out + (0 .. na + nb - 2))`. Requirements: implement `poly_multiply(const int64_t *a, int na, const int64_t *b, int nb, int64_t *out)` so that for every k in [0, na + nb - 1), `out[k] == spec_poly_mul_coeff(a, na, b, nb, k)`. (1) Algorithm: zero-initialize out[0 .. na+nb-2], then for every i in [0, na) and every j in [0, nb), update `out[i + j] += a[i] * b[j]`. (2) Functional Correctness: the postcondition uses `spec_poly_mul_coeff(a, na, b, nb, k)`, which is defined via `spec_convolution_sum(a, na, b, nb, k, i)` recursing on i — base case 0 when i < 0, step case `((0 <= i < na && 0 <= k - i < nb) ? a[i] * b[k - i] : 0) + spec_convolution_sum(.., i - 1)`. Verification Challenges: (1) Accumulation invariant — the same out[k] is updated multiple times across the (i, j) iteration, so the loop invariant must say "after processing (i, j), out[i + j] equals the partial convolution sum restricted to (i', j') already visited." A clean form is to let the outer-loop invariant assert `out[k] == sum over (i', j') with i' < i and i' + j' == k of a[i']*b[j']` and refine it inside the inner loop. (2) Overflow proof — each product a[i]*b[j] fits in i64 (|a[i]*b[j]| <= 10^12), and there are at most min(k+1, 1000) such products contributing to out[k], giving |out[k]| <= 10^15 < i64::MAX = 2^63 - 1 ≈ 9.2 * 10^18. The proof obligation must capture this bound as a numeric invariant. (3) Linking to spec_convolution_sum — the iteration order in the algorithm differs from the recursion order of the spec; you'll need a ghost lemma (kept inside the function body) that re-indexes the partial sum to align with the spec_convolution_sum_step axiom. ACSL-specific notes: (a) The contract requires `\separated(a + (0 .. na - 1), b + (0 .. nb - 1), out + (0 .. na + nb - 2))` since na + nb might be small but nothing in the C signature prevents a, b, and out from aliasing — Verus's borrow-checker provides this implicitly via `Vec` ownership, but in ACSL the writes to out can invalidate a/b reads without an explicit non-alias precondition. (b) Use `loop variant na - i` (resp. nb - j) for termination. (c) Do not introduce free-standing axioms about spec_convolution_sum; rely on the supplied step axiom and unfold ghost-style inside the inner loop. (d) `-wp-model Typed` plus `\separated` is the right setup; the integer theory dispatches the bound proofs once the numeric invariant is recorded.
