Your task is to implement the Zig-Zag (left-right rotation) operation on a Binary Search Tree and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: The input pointer `g` is valid for reading and satisfies `is_bst(g)`, with `g->left != \null`, `g->left` valid for reading, and `g->left->right != \null`. The Zig-Zag case applies when grandchild `X = g->left->right`, parent `P = g->left`, and grandparent `G = g` form a "kink" (`X` is the right child of `P`, `P` is the left child of `G`). Requirements: Implement `bst_zig_zag(BstNode *g)` returning a non-null `BstNode *` whose root value equals the original `g->left->right->val` (`X` becomes the new root, with `P` as left child and `G` as right child). Pictorially, given `g = (G, P = (P, A, X = (X, B, C)), D)`, the result is `(X, (P, A, B), (G, C, D))`. (1) Algorithm: This is two single rotations: first a left-rotation at `P` (turning `(P, A, (X, B, C))` into `(X, (P, A, B), C)`) and then a right-rotation at `G` (turning `(G, (X, ...), D)` into `(X, ..., (G, ..., D))`). Equivalently, allocate (or re-link) so that `X` is at the top with `(P, A, B)` on the left and `(G, C, D)` on the right. (2) Functional Correctness: Postconditions: `\result != \null`, `is_bst(\result)`, view-equality `\forall integer x; in_view(\result, x) <==> in_view{Pre}(g, x)`, and `\result->val == \at(g->left->right->val, Pre)`. The preamble provides `in_view` and `is_bst` (recursive predicates). Verification Challenges: (1) BST invariant preservation: in the rotated tree the root is `X`, the left child is `(P, A, B)`, and the right child is `(G, C, D)`. The proof requires showing: every value in `A` is `< P` (already true), every value in `B` is `> P` and `< X` (because `B` was the left subtree of `X` in the original, hence `< X`; and `B` was inside the right subtree of `P`, hence `> P`), every value in `C` is `> X` and `< G`, and every value in `D` is `> G`. (2) Set Preservation: re-parenthesization preserves the multiset. (3) X "splits" its original children: the trickiest part is that `X`'s subtrees `B` and `C` are sent to *different* places (`B` becomes the right child of `P`, `C` becomes the left child of `G`), so the case split must be done explicitly in ghost code. ACSL-specific notes: (a) The contract is `assigns \nothing;` — the implementation allocates fresh nodes for the new root and the new `P`, `G` nodes, leaving the original tree intact. Choose this functional style to keep `assigns` small. (b) `\at(g->left->right->val, Pre)` captures `X`'s value at entry. (c) `is_bst` is recursive; supply `decreases` on call sites or invoke it on subtrees of known ghost-height. (d) The default `-wp-model Typed` memory model is sufficient. (e) Use ghost bindings `X_v`, `P_v`, `G_v`, and ghost subtree pointers `A`, `B`, `C`, `D` at function entry, then explicitly assert the BST ordering between subsets of those before/after the rebuild — `assert (\forall integer y; B != \null && in_view{Pre}(B, y) ==> P_v < y < X_v);` etc. (f) Per project rules, no top-level lemmas or axioms in the user solution.
