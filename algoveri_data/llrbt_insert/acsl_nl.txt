Your task is to implement and verify the insertion algorithm for a Left-Leaning Red-Black Tree (LLRBT) in ACSL using Frama-C/WP. Preconditions: `t` is either `\null` or a valid `LlrbtNode *` satisfying `is_llrbt(t)` (i.e. `is_bst(t) && black_height(t) != -1 && satisfies_red_props(t)`); the helper predicate `opt_is_llrbt` factors this conditional. Requirements: Implement `llrbt_insert` returning a non-null `LlrbtNode *`. (1) Algorithm: Standard recursive BST insertion adding new nodes as Red, followed by, on the unwinding phase, the three LLRBT fixups in a fixed order: rotate left if right child is Red and left child is not, rotate right if left child is Red and its left child is also Red, flip colours if both children are Red. (2) Functional Correctness: The contract requires `\result != \null`, `is_llrbt(\result)`, and `in_view(\result, x) <==> (x == v || (t != \null && in_view{Pre}(t, x)))` (the new view is the original view with `v` inserted). STRUCTURAL DEFECT (flagged challenge): The contract states `assigns \nothing;` while requiring a non-null result whose view differs from the input. If `t == \null`, returning a non-null result requires fresh heap allocation; if `t != \null`, structural rebalancing requires either fresh allocation of internal nodes or in-place mutation of the existing tree. Both options conflict with `assigns \nothing;`. As written, the spec is unsatisfiable in standard WP semantics. The recommended fix is either (a) replace `assigns \nothing;` with `assigns \nothing; allocates \result;` and stipulate that the new tree's nodes are fresh (heap-allocated, no aliasing with the old tree, which then leaks), or (b) declare `assigns *(LlrbtNode *)t, *(LlrbtNode *)t->left, ...;` reflecting the in-place rotation/recolouring updates, with explicit ghost reasoning about the reachable footprint. Both approaches are heavyweight; this is a known limitation of expressing functional-style tree algorithms in C/ACSL. Verification Challenges: (1) Structural Defect (above) - blocks any sound implementation. (2) Recursive Verification: Insertion is naturally recursive; you must specify each recursive call as a contract preserving `is_llrbt` weakly (the result may have a Red root, which `is_llrbt` permits) and the view-update postcondition. (3) Three-Way Fixup Soundness: After each fixup primitive (`rotate_left`, `rotate_right`, `flip_colors`), you must show the partial invariants are restored. Ideally these are imported as separate verified lemmas / functions; the present file does not import them, so each fixup must be inlined with its own correctness argument. (4) Black-Height Inductive Invariant: Insertion preserves `black_height` exactly (the new node is Red, contributing 0 to the height); the proof depends on combining the height axiom with the colour fixups - a non-trivial inductive argument over the recursion depth. ACSL-specific notes: (a) The `assigns \nothing;` clause is the load-bearing defect; one practical workaround is to add an `allocates \fresh_tree;` hypothesis or weaken `assigns` to cover the modified pointer fields of the entire reachable footprint of `t`. (b) The `is_llrbt` predicate combines three sub-predicates; you should likely break the postcondition apart and discharge each separately. (c) The `axiomatic LlrbtBlackHeight` block declares `reads t, t->is_red, t->left, t->right;` - one level - so unfolding the height across multiple recursive levels requires manual lemmas. (d) Aliasing: ACSL has no built-in acyclicity for trees; ghost separation properties between subtrees may be necessary for the in-place version. (e) `-wp-model Typed` is sufficient for the property reasoning, but if you switch to a heap-allocating spec, `-wp-model Typed+ref` may help reason about freshness. (f) No top-level lemmas required by contract, but at minimum lemmas for each fixup primitive's effect on `is_bst`, `black_height`, and `satisfies_red_props` are required as ghost code.
