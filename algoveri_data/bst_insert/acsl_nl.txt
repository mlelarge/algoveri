Your task is to implement Binary Search Tree insertion and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: The input is a (possibly null) pointer `t` to a `BstNode` satisfying `opt_is_bst(t)`. The value `v` is a `uint64_t`. Requirements: Implement `bst_insert(BstNode *t, uint64_t v)` returning `BstNode *` — a non-null pointer to the root of the new tree. The implementation may either build a fresh tree with allocation (`malloc`) or rebuild via the standard recursive functional pattern. (1) Algorithm: If `t == NULL`, allocate a fresh leaf with value `v` and `left/right == NULL`. Otherwise, compare `v` to `t->val`: if `v < t->val`, recursively insert into the left subtree; if `v > t->val`, recursively insert into the right subtree; if equal, return `t` unchanged (sets are idempotent under reinsertion). Return the (possibly new) root. (2) Functional Correctness: The postconditions are `\result != \null`, `is_bst(\result)`, and `\forall integer x; in_view(\result, x) <==> (x == v || (t != \null && in_view{Pre}(t, x)))`. The preamble provides `in_view`, `is_bst`, and `opt_is_bst` (described in the search problem). Verification Challenges: (1) BST invariant preservation: when descending into a subtree, after the recursive insert returns, you must show that every value in the new subtree still respects the parent ordering — i.e., if you inserted into the left subtree, every value there (including `v`) is still `< parent->val`. (2) View update: the view of the new tree equals the old view union `{v}`. The recursive case unfolds the view of the parent tree as `{t->val} ∪ view(left) ∪ view(right)`, where exactly one of `left` or `right` is replaced by the recursive result whose view is `view(child) ∪ {v}`. ACSL-specific notes: (a) The contract is `assigns \nothing;` even though the implementation typically allocates new nodes — ACSL's `assigns` clause restricts only writes to *pre-existing* memory. Allocation creates new memory cells outside the `Pre` heap, so `assigns \nothing` and `malloc` coexist (Frama-C's WP supports `\fresh(\result)` reasoning under the default memory model). (b) `in_view{Pre}(t, x)` evaluates the predicate at the `Pre` label so the postcondition refers to the *original* tree's view, not whatever `t` now points to. Be careful when iteratively descending: capture `\at(t, Pre)` before mutation. (c) `in_view` and `is_bst` are recursive predicates; supply a `decreases` measure (e.g., a ghost-tracked tree height) for each recursive call. (d) The default `-wp-model Typed` memory model is sufficient; if `malloc` causes heap-fragmentation worries, switch to `-wp-model Typed+ref`. (e) For the iterative implementation, maintain an invariant relating each visited node's path to the structural decomposition of `in_view`. (f) Per project rules, no top-level lemmas or axioms in the user solution — extend reasoning via `assert` and `loop invariant` only.
