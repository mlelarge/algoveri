Your task is to implement the Coin Change algorithm (minimum number of coins to make a target amount) and verify its functional correctness in ACSL using Frama-C/WP. Preconditions: The input `coins` is a `const uint64_t *` of length `n` with `0 < n <= 100`, every `coins[i]` lies in `(0, 10000]`, and `amount` is a `uint64_t` bounded by `10000`. Requirements: Implement `int64_t coin_change(const uint64_t *coins, int n, uint64_t amount)`. Return the minimum total coin count over every non-negative `counts[0..n)` whose weighted sum `total_amount(counts, coins, 0, n) == amount`, or `-1` if no such combination exists. Algorithm: Use bottom-up dynamic programming on `dp[0..amount]`, where `dp[a]` is the minimum coin count summing to `a` (or a sentinel meaning "unreachable"). For each `a` from `1` to `amount`, take `dp[a] = 1 + min over i with coins[i] <= a of dp[a - coins[i]]`. Verification Challenges: (1) Existence: when `\result != -1` you must exhibit a witness `int *counts` with `is_valid_change(counts, coins, n, amount)` and `total_coins(counts, 0, n) == \result`. The natural witness is the counts vector you reconstruct by walking back through `dp`. (2) Lower Bound: every valid combination satisfies `total_coins(...) >= \result`; this needs a loop invariant tying `dp[a]` to "minimum over all valid count vectors of length n summing to a". (3) Impossibility: when `\result == -1`, no valid combination exists at all. ACSL-specific notes: (a) The axiomatic block `CoinChangeSums` defines `total_amount` and `total_coins` recursively with base/step axioms keyed at index `k == n`; you should drive proofs by induction on `k` using ghost lemmas applied inside loops rather than top-level lemmas (the spec file has none on purpose). (b) The ensures clauses use `\exists int *counts, ...` and `\forall int *counts, ...`; the WP `Typed` model treats these as quantification over phantom buffers, so you must witness or refute existence with a concrete counts pointer (typically a ghost array). (c) Although `coins[i] * counts[i]` could overflow a u64 for adversarial inputs, the bounds (`coins[i], amount <= 10000`, `n <= 100`) make `total_amount` fit in 64 bits comfortably; nevertheless work in `integer` math inside the contract, as the axiomatic already does. (d) `assigns \nothing` since the function only reads. (e) Provide a loop variant on the outer DP loop (e.g. `amount - a`) for termination. (f) Default `-wp-model Typed` plus Alt-Ergo/Z3 should suffice; no top-level lemmas are required, only inline ghost code.
